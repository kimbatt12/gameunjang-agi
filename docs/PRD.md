## 1. 제품 개요

**Gameunjang-agi**는 한국관광공사 공공데이터 API를 활용하여 국내 관광 관련 질문에 답변하는 모바일 우선 웹 챗봇이다. 사용자는 국내 여행을 계획하는 내국인 관광객이며, 관광지·행사·숙박·음식점·여행코스·지역 정보 등을 질문하면 공공데이터 기반으로 추천과 설명을 받을 수 있다.

제품은 월 운영비 **1달러 미만**을 목표로 하며, 일 1,000쿼리 미만의 사용량을 기준으로 설계한다. 답변 생성에는 LLM API를 사용하며, 기본 LLM provider는 **Upstage API**로 결정한다. OpenRouter API는 어댑터 기반 대체 provider로 지원한다. 호스팅은 **Vercel**로 결정한다.

## 2. 목표

### 2.1 핵심 목표

1. 국내 관광 질문에 대해 신뢰 가능한 답변 제공
2. 한국관광공사 공공데이터 API 전체를 후보 도구로 활용
3. 질문 의도를 판단하여 적절한 API를 의미검색 기반으로 선택
4. 관광 관련 질문이 아니면 답변을 거절
5. 운영비 월 1달러 미만 유지
6. 모바일 우선, PC 반응형 웹서비스 제공
7. API 명세/메타데이터 변경을 주 1회 자동 갱신 및 수동 갱신 가능하게 관리
8. 갱신 실패 시 기존 정상 데이터 유지

### 2.2 비목표

초기 MVP에서는 아래를 목표로 하지 않는다.

- 사용자 로그인
- 서버 측 대화 기록 저장
- 커뮤니티 기능
- 결제 기능
- 관리자 대시보드 내 실시간 상세 모니터링
- 네이버/카카오/블로그 등 비공식 출처 기반 답변
- 월 1달러를 초과하는 고성능 벡터 DB 또는 상용 검색 인프라 의존

## 3. 대상 사용자

### 3.1 주요 사용자

- “제주도 비 오는 날 갈 만한 곳 알려줘”

- “이번 달 서울 축제 알려줘”

## 4. 핵심 사용자 시나리오

### 4.1 국내 관광 질문 응답

1. 사용자가 챗 화면에 질문 입력
2. 시스템이 LLM scope classifier로 질문이 국내 관광 관련인지 판별
3. 관련 없는 질문이면 정중히 거절
4. 관련 질문이면 API 메타데이터와 라우팅 신호로 사용할 API 후보 선정
5. 한국관광공사 API 호출
6. 필요 시 허용된 공식/공공 출처에서 보완 정보 사용
7. 답변 생성
8. 답변 하단에 출처 도메인 표시

### 4.2 일정 추천

사용자가 “2박 3일 강릉 여행 코스 짜줘”처럼 일정 추천을 요청하면 시스템은 관광지, 음식점, 숙박, 축제/행사, 여행코스 API를 조합하여 일자별 추천 일정을 생성한다.

### 4.3 비관광 질문 거절

사용자가 국내 관광과 무관한 질문을 입력하면 시스템은 LLM scope classifier 판별 후 답변 생성을 중단하고 다음과 같이 응답한다.

> 이 서비스는 국내 관광 관련 질문에만 답변할 수 있습니다. 여행지, 관광지, 축제, 숙소, 음식점, 여행코스와 관련된 질문을 입력해 주세요.

## 5. 기능 요구사항

## 5.1 챗봇 UI

### FR-001: 모바일 우선 챗 화면

- 사용자는 모바일 브라우저에서 질문을 입력할 수 있어야 한다.
- PC에서도 반응형으로 사용할 수 있어야 한다.
- 초기 UI는 단순한 대화형 화면으로 구성한다.

### FR-002: 로컬 대화 기록

- 대화 기록은 서버에 저장하지 않는다.
- 브라우저 localStorage 또는 IndexedDB에 저장한다.
- 사용자는 같은 브라우저에서 이전 대화를 볼 수 있다.
- 사용자는 로컬 대화 기록을 삭제할 수 있어야 한다.

### FR-003: 브라우저 세션당 대화 제한

- 비용 통제를 위해 브라우저 세션당 최대 10회까지 질문할 수 있다.
- 서버 계정/로그인이 없으므로 제한 기준은 브라우저 세션 단위로 한다.
- 제한에 도달하면 새 세션을 시작하거나 대화를 초기화하라는 안내를 표시한다.

---

## 5.2 질문 분류

### FR-004: 국내 관광 관련성 판단

시스템은 답변 생성 전에 LLM scope classifier로 질문이 국내 관광 관련인지 판단해야 한다. classifier는 `domestic_tourism` 또는 `out_of_scope`의 엄격한 label만 반환한다.

관광 관련으로 허용되는 예:

- 국내 관광지
- 국내 축제/행사
- 숙박
- 음식점/맛집
- 여행코스
- 지역 기반 추천
- 아이와 갈 만한 곳
- 실내/실외 관광
- 날씨를 고려한 관광지 추천
- 국내 여행 일정
- 관광지 주소/운영시간/비용/위치/공식 링크 문의

거절해야 하는 예:

- 해외 여행
- 정치/연예/코딩/투자/일상 상담 등 국내 관광 무관 질문
- 국내 지역명이 포함되더라도 관광 의도가 없는 질문

### FR-005: 거절 응답

- 국내 관광과 무관하면 LLM scope classification 이후 외부 검색/API 호출 없이 정중히 거절한다.
- scope classifier provider가 없거나 사용할 수 없거나 응답 label이 올바르지 않으면 안전한 범위 안내로 종료한다.
- 거절 응답은 짧고 명확해야 한다.

## 5.3 API 활용

### FR-006: 한국관광공사 API 전체 후보화

- 공공데이터포털에서 제공되는 한국관광공사 관련 API를 전부 수집 대상으로 삼는다.
- 모든 API를 매 질문마다 호출하지 않고, 질문에 맞는 API 후보를 검색하여 호출한다.
- 초기 API 키는 하나의 `TOUR_API_SERVICE_KEY`로 관리한다.

### FR-007: API 메타데이터 저장

시스템은 각 API에 대해 다음 정보를 저장해야 한다.

- API 이름
- 설명
- 제공 기관
- 엔드포인트
- 주요 파라미터
- 필수 파라미터
- 응답 필드
- 데이터 범위
- 사용 예시
- 출처 도메인
- 마지막 성공 갱신 시각

### FR-008: 의미검색 기반 API 선택

- LLM scope classifier가 `domestic_tourism`으로 수락한 질문을 API 메타데이터와 비교하여 관련 API 후보를 선택한다.
- 초기에는 정적 JSON 인덱스 + 키워드/간단 의미검색 기반 라우팅 메타데이터를 사용해 Tourism/KMA API 후보를 고른다.
- 향후 필요 시 임베딩 인덱스를 추가할 수 있다.

## 5.4 답변 생성

### FR-009: API 데이터 우선 답변

- 답변은 한국관광공사 API 데이터를 우선 사용해야 한다.
- API 데이터로 확인되지 않은 정보는 추측하지 않는다.

### FR-010: 보완 출처 사용

API 데이터에 없는 정보는 한국관광공사 API 응답에 포함된 공식 링크에서만 보완할 수 있다.

허용 출처:

- 정부기관
- 지자체
- 공공기관
- 관광지 공식 홈페이지
- `.or.kr` 도메인
- `.go.kr` 도메인

금지 또는 기본 제외 출처:

- 개인 블로그
- 카페
- 광고성 페이지
- 네이버/카카오 일반 검색 결과
- 출처 신뢰도를 확인하기 어려운 페이지
- API 응답에 포함되지 않은 임의 웹 검색 결과

### FR-011: 답변에 포함할 정보

가능한 경우 답변에는 다음을 포함한다.

- 추천 목록
- 추천 이유
- 위치/주소
- 운영시간
- 비용
- 이동 동선
- 공식 링크
- 지도 링크
- 날씨 고려 사항
- 출처 도메인

### FR-012: 출처 표시

- 답변 하단에 출처 도메인을 표시한다.
- 형식 예:

```text
출처: visitkorea.or.kr, data.go.kr, gangneung.go.kr
```

### FR-013: 정보 없음 처리

- API와 허용 출처에서 확인할 수 없는 내용은 “확인된 정보가 없습니다”라고 표시한다.
- 불확실한 정보를 단정하지 않는다.

## 5.6 날씨 고려

### FR-016: 날씨 API 연동

- MVP에서 외부 날씨 API 연동을 포함한다.
- 날씨 API는 기상청 API를 기준으로 한다.
- 비용이 없는 공공/무료 사용 범위를 우선한다.
- 기상청 API 사용에 필요한 지역 좌표/격자 변환을 고려한다.

### FR-017: 날씨 기반 추천

- 비/눈/폭염/한파 등 조건에 따라 실내 관광지 또는 안전한 코스를 우선 추천한다.
- 날씨 정보 출처 도메인을 답변 출처에 포함한다.

## 5.7 데이터 갱신

### FR-018: 주 1회 자동 갱신

- API 메타데이터와 사용법은 주 1회 자동 갱신한다.
- GitHub Actions cron을 우선 추천한다.

### FR-019: 수동 갱신

- 관리자는 GitHub Actions `workflow_dispatch`로 수동 갱신할 수 있다.
- 웹앱 내 관리자 버튼은 MVP에서 제외한다.
- 관리자는 사용자 본인 1명만 고려한다.

### FR-020: 원자적 갱신

- 갱신 작업 전체가 성공했을 때만 새 API 메타데이터로 교체한다.
- 하나라도 실패하면 기존 정상 메타데이터를 유지한다.
- 갱신 실패 알림은 MVP에서 제공하지 않는다.

## 6. 비기능 요구사항

### NFR-001: 비용

- 전체 운영 비용은 월 1달러 미만을 목표로 한다.
- 비용 범위에는 호스팅, DB, LLM, 배치, 저장소가 포함된다.
- 무료 티어를 최대한 활용한다.

### NFR-002: 벤더 종속 최소화

- 호스팅은 Vercel로 결정하되, 핵심 도메인 로직은 Vercel 전용 API에 강하게 결합하지 않는다.
- 특정 DB, 전용 런타임, 전용 언어에 강하게 묶이지 않는다.
- 비용이 같거나 더 저렴하다면 벤더 종속이 낮은 구현 방식을 우선한다.
- React 프론트엔드 + Python 백엔드처럼 필요 시 다른 호스팅으로 이전 가능한 구성을 선호한다.

### NFR-003: 호스팅

- MVP 호스팅 플랫폼은 Vercel로 결정한다.
- Vercel 무료 티어를 우선 사용한다.
- 프론트엔드 정적 배포와 Python 기반 API 라우트를 Vercel에서 운영하는 구성을 우선 검토한다.
- Vercel 전용 구성 파일은 배포 어댑터로 격리하고, 핵심 비즈니스 로직과 데이터 처리 로직은 플랫폼 독립적으로 유지한다.

### NFR-004: 성능

- 일반 질문 응답은 1~3초 이내를 목표로 한다.
- 복잡한 일정 추천은 더 오래 걸릴 수 있으나 사용자에게 처리 중 상태를 표시한다.

### NFR-005: 개인정보

- 로그인 없음
- 서버 측 대화 기록 저장 없음
- 브라우저 로컬 저장만 사용
- 민감정보 입력을 권장하지 않는다.

### NFR-006: 신뢰성

- 갱신 실패 시 기존 데이터 유지
- API 호출 실패 시 부분 결과 또는 확인 불가 응답 제공
- 출처 없는 외부 정보 사용 금지

## 8. 성공 지표

1. 국내 관광 질문에 대해 관련 API를 선택해 답변할 수 있다.

3. 답변 하단에 출처 도메인을 표시한다.
4. 일 1,000쿼리 미만 기준 월 1달러 미만 운영 가능성이 유지된다.
5. 주 1회 갱신이 성공 시에만 배포 데이터에 반영된다.
6. 모바일에서 사용 가능한 단순 챗 UI가 제공된다.

## 9. 주요 리스크

### R-001: LLM API 비용 및 공급자 종속성

Upstage API를 기본 provider로 사용하고 OpenRouter API를 대체 provider로 지원하되, scope classification과 답변 생성에 LLM을 사용하므로 사용량 증가에 따라 월 1달러 미만 목표를 초과할 수 있다. 또한 특정 LLM 공급자의 장애, 가격 변경, 모델 변경에 영향을 받을 수 있다.

대응:

- LLM 호출부를 추상화한다.
- Upstage, OpenRouter, 향후 다른 유료 모델로 교체 가능하게 설계한다.
- 캐싱과 템플릿 답변으로 LLM 의존도를 낮춘다.
- 브라우저 세션당 10회 대화 제한을 적용한다.
- scope classifier는 작은 prompt와 낮은 `max_tokens`로 운영한다.
- provider/fallback 설정으로 공급자 장애 영향을 줄인다.

### R-002: 월 1달러 미만 비용

모든 질문에 scope classification LLM을 호출하고 수락된 질문에 답변 생성 LLM을 사용하는 전략은 비용 초과 가능성이 있다.

대응:

- 강한 캐싱
- 중복/유사 질문 재사용
- 소형 모델 우선
- API 결과 기반 템플릿 답변 보조
- 브라우저 세션당 10회 질문 제한

### R-003: API 명세 다양성

한국관광공사 API가 여러 묶음과 다양한 파라미터를 가질 수 있다.

대응:

- API 메타데이터 인덱스를 별도 생성
- API별 호출 어댑터 분리
- 갱신 성공 시에만 교체

### R-004: 출처 제한으로 답변 범위가 좁아질 수 있음

허용 도메인을 정부/지자체/공공기관/관광지 공식 홈페이지로 제한하면 일부 최신 정보가 부족할 수 있다.

대응:

- 정보 없음 응답 허용
- 출처가 없는 정보는 단정하지 않음
